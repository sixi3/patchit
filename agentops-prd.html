Brief

This product should not be another chat UI. It should be the operational layer where software engineers manage agentic work: assignment, status, quality, risk, cost, and decisions.

Version 0.1

Problem

Developers can now start coding agents from many places, but supervising the work is fragmented. Each run creates review debt, context debt, and cost uncertainty.

Product Definition

A mobile-first AgentOps cockpit for software teams: route work to agents, monitor execution, review outputs, and approve next actions.

Initial Bet

The first valuable loop is not writing code from a phone. It is trusting a decision about agent-generated work from a phone.

Positioning Line

AgentOps for software engineers: one developer-native command center to dispatch, monitor, review, and budget AI coding agents.

Non-Sycophantic Take

The idea is strongest as a supervision and review layer. It is weakest if it becomes a generic wrapper around Claude, ChatGPT, and GitHub. The product must create new software-specific decision surfaces, not just aggregate chats.

User

Start with engineers who already feel agent sprawl. They are not asking whether agents work; they are asking which ones to trust, where work stands, and what needs human judgment.

Job To Be Done

When I have multiple agent tasks running, I want to know what is ready, risky, blocked, or wasteful, so I can make the next engineering decision without opening my laptop.

Pain Today

  • Run state lives in separate tools.
  • PR summaries are not decision-grade.
  • Costs are opaque until after the run.
  • Mobile review is awkward and diff-first.

Trigger Moment

The user has more than two active agent runs and cannot quickly answer which one is good, stuck, expensive, or ready to merge.

Buyer Shape

Individual prosumer first. Team product later after the workflow proves repeat usage and trust.

First Workflow

Keep the first release narrow and complete: take a GitHub issue, route it to an agent, monitor execution, summarize the PR, and let the engineer make a decision.

Decision Grammar

Each task should collapse into one recommended next action: approve, inspect, fix tests, ask human, rerun, compare, or stop.

Review Surface

Start with intent alignment, behavior change, risk, tests, touched files, and screenshots. Raw diff is a drill-down, not the front door.

Notification Rule

Notify only when a task changes decision state. Running logs should not become a new attention tax.

Risks

The riskiest parts are not UI polish. They are trust, API access, execution reliability, and avoiding a shallow aggregator.

Reality Check
High

Native harnesses close the gap

Claude, Codex, GitHub, and IDE vendors can add better mobile supervision. Mitigation: own cross-agent workflows, cost routing, and review grammar.

High

Integration limits

Some proprietary harnesses may not expose full lifecycle control. Mitigation: build the deepest experience on OpenCode/self-hosted runners first and use lighter connectors elsewhere.

Medium

Developers distrust mobile merge decisions

Mitigation: start with low-risk tasks, excellent summaries, explicit risk flags, and easy escalation to desktop/GitHub.

Medium

Cost estimates are noisy

Mitigation: frame cost as a range, compare historical routes, and enforce budget caps instead of promising exact token forecasts.

Low

Too many inboxes

Mitigation: the app should replace agent status checking, not become another place for general team chat.

Activation

User connects GitHub, imports an issue, dispatches one agent task, and reviews the resulting PR summary.

Retention

User returns because the app tells them what agent work needs a decision before GitHub, Slack, or terminal spelunking does.

Willingness To Pay

Users pay if the product saves senior engineering attention, prevents wasted runs, or lets them route work to cheaper capable models.

Open Questions

Which harness exposes enough control first? How much code review can be safely mobile-native? Is cost routing a primary wedge or retention feature?

Recommended MVP Claim

Review, steer, and approve agent-generated PRs from your phone, with cost-aware routing across coding agents.
