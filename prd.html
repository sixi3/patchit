Loupe

One inbox for tickets, running agents, and the PRs they raise — built for engineers who direct two or more coding agents a day.

Date2026-05-25

TL;DR

A senior engineer in 2026 has work entering through JIRA, executing on Claude Code and Codex , and landing as PRs on GitHub — across at least three surfaces, none of which know about the others. Loupe collapses that into one mobile inbox: tickets, running agents, and agent-authored PRs in a single stream, tagged by type and sorted by attention need.

The wedge isn't dispatch — vendors and Atlassian Rovo already do that. The wedge is the review surface for agent-authored PRs: a layered, question-driven form factor where the agent commits to a recommendation, and cross-agent critique (one agent plans, another reviews) is built into the dispatch flow.

Design language: Linear's tight information density meets PagerDuty's priority-driven, action-oriented inbox.

Naming

The product is Loupe. A loupe is the magnifying glass jewelers and photographers use to inspect detail — semantically aligned with the wedge (close inspection of agent work). Short, unusual, low-collision in dev tooling. Prior associations (Loupe photography app, jeweler-tool brands) sit outside our market and don't fight us for SEO or discovery. Trademark search before launch; no obvious blockers.

Competitive density

Cursor 3 "Glass" (Apr 2026) ships an Agents Window with multi-source, multi-LLM management. Atlassian Rovo (Feb 2026) ships AI agents inside JIRA with MCP-compatible third-party agents. Anthropic and OpenAI have shipped Dispatch and Codex Mobile. Conductor (Melty Labs) does parallel agent runs on macOS. Loupe's only durable wedge is the PR review surface and the cross-agent critique loop — the places all four leave open.

Problem

  1. Work has too many homes. Tickets in JIRA, agents in two terminals, PRs on GitHub mobile. None share context.
  2. PR review was designed for human-to-human handoff. Diffs assume shared context. With agents, the reviewer is meeting the work cold. Most agent failures live in misread intent or unflagged approach choices — neither visible in the diff.
  3. Throughput, not thoroughness, is now the constraint. A senior eng went from 2 PRs/day to 20. Existing review tools optimize for the wrong axis.
  4. Multi-agent critique is a manual workflow. Power users paste a plan from one CLI into another for a sanity check. It works, but it's a copy-paste tax. Nobody has productized the loop.

Target user

Senior+ engineers at JIRA-shop companies who already run Claude Code , Codex , or both daily. BYO API keys. Review ≥1 agent-authored PR a day. Individual purchaser. Not teams (yet), not policy buyers, not non-engineers.

Design inspiration

Linear — information density

What to steal

  • Tight rows, monospace metadata, near-monochromatic palette with a single accent.
  • Type tags as small pills, not big badges. The data is the design.
  • Cmd+K as the primary action surface; on mobile, a persistent compose affordance.
  • Inbox is one stream with type filters, not multiple lists.
  • Read/unread state as a primary visual primitive.
PagerDuty — priority & action

What to steal

  • Priority-color stripe on the left edge of every row. Triage state in one glance.
  • Acknowledge / Resolve / Escalate verbs always present. Each row knows its actions.
  • "On it" state — show when something is already being handled.
  • Snooze affordance: not "delete," but "not now."
  • Severity scales (P1–P4) translate to risk levels (HIGH / MED / LOW / routine).

The synthesis: a Linear-tight inbox where each row carries a PagerDuty-style priority stripe, a type tag, and a single dominant action.

Core concept · cross-agent critique

Power users have already discovered this workflow: ask Codex to draft a plan; paste it into Claude for review; iterate; then dispatch. It works for three structural reasons:

  • Different harnesses have orthogonal priors. Errors one would miss, the other catches.
  • The critic isn't invested. It didn't write the plan, so it has no confirmation bias to defend.
  • Plans are cheap. Killing a bad plan costs $0.05; killing a bad implementation costs $0.50 + 15 minutes.

Loupe productizes this loop. It appears as an opt-in toggle under the "Plan & dispatch" mode in the dispatch sheet. When enabled: planner emits the plan, critic reviews it, you see both in a single annotated document, you approve (with edits if needed), the implementer codes with both plan + accepted critique as first-class context.

This is also the v1.1 path for the PR review surface: when a ticket was parallel-dispatched, the recommendation card becomes a meta-review — *where the two agents agree*, you skim; *where they disagree*, you spend attention. Disagreement is the highest-signal element in the whole product.

2 · Dispatch sheet

Opens when the user taps a ticket. Three modes. When Plan & dispatch is selected, a second-opinion toggle becomes available. Then agent. Then fire.

The three modes

  • Dispatch — agent goes immediately with the ticket as brief. For routine, well-scoped work.
  • Plan & dispatch — agent emits a markdown plan first. You approve or edit. Only then does it code. Optional sub-feature: second opinion, where a different agent reviews the plan before you approve.
  • Send to me — mark the ticket as mine. Mutes in Loupe, leaves the ticket in JIRA. Loupe never becomes a to-do list.

3 · Plan review · two-agent critique

The pivot when Plan & dispatch is used. Default tab is Combined: plan and critic comments interleaved, each comment pinned to the section it's about. Adopt or dismiss each note individually — adopted notes become first-class constraints for the implementer; dismissed notes don't.

This is where intent misreads die at minute 2 instead of minute 9. The agent that didn't write the plan is the one most likely to catch what the planner missed — it has no skin in the plan being right.

5 · PR review · the wedge

Built on three composable ideas:

  • Layered briefing — the agent commits to a recommendation at the top. Confidence + the single thing worth reading. The rest is opt-in depth.
  • Question-driven body — instead of seven prose sections, the body is a list of questions a reviewer would ask. Each row is yes/no/unknown with one-tap expansion to evidence.
  • Risk-weighted diff entry — when the user opens code, they enter through risk, not file tree.

Reviewers optimize confidence per minute spent. The diff is uniform — every line equal visual weight. The recommendation + question list collapses the surface into a *scannable decision* on mobile, with depth available on demand.

v1.1 · cross-agent meta-review at PR stage

Same pattern as plan critique, applied to the PR: implementer (Agent A) writes the code and emits its self-assessment. Critic (Agent B), seeing only the diff and the original intent, emits its own read. The recommendation card becomes a meta-review — where they agree, you skim; where they disagree, you spend attention. Disagreement is the highest-signal element in the entire product.

JIRA integration

  • Atlassian OAuth 2.0 (JIRA Cloud only v1). Pull issues assigned to user across visible projects.
  • Configure per-project repo binding in settings. Loupe passes the repo context at dispatch.
  • Auto-dispatch trigger: JIRA label loupe:auto applied opt-in by the user.
  • Round-trip: comment on the source ticket when PR opens; on accept, transition per-project rule (default: don't transition).

BYO keys & spend

  • User pastes Anthropic + OpenAI API keys in settings. Keys stay in the Mac bridge daemon's keychain; not synced to the phone.
  • Aggregated spend (this month, this hour burn rate) on the inbox summary strip. Not "remaining balance" — providers don't expose that cleanly for BYO keys; we track usage ourselves.
  • Per-task cost shown at dispatch (heuristic on day 1, historical-similar from week 2).
  • Per-PR actual cost shown in the form factor's submeta.
  • No auto-routing in v1. User picks.

Architecture (minimal)

  • iOS app (SwiftUI) — surfaces above.
  • macOS bridge daemon — holds keys in keychain, invokes Claude Code + Codex CLIs locally, watches GitHub, polls JIRA. Takes PreventUserIdleSystemSleep while an agent runs.
  • Transport — WebSocket between phone and bridge, paired via QR code; X25519/Ed25519/AES-256-GCM.
  • Cloud relay (small) — only for: APNS push to wake the phone, JIRA webhook landing pad. Stateless.
  • State — bridge owns truth; phone is a thin client with local cache for offline read-only.
  • Form-factor generation — bridge asks the harness for structured output where possible, post-processes with a cheap LLM call to fill recommendation + question list + risk flags. Hybrid. Needs a spike.

Build order

  1. Bridge daemon + auth. macOS daemon, QR pair to phone, BYO key storage, can shell out to Claude Code with a prompt.
  2. Dispatch + Live view. Free-form task from phone → agent runs locally → steps stream back.
  3. Form-factor generation pipeline. Recommendation + question list generation from real PRs. Deepest unknown — spike first. Includes confidence-calibration validation.
  4. PR review surface. Render the form factor. Wire 4-verb actions to GitHub.
  5. Inbox. Real data, type filters, swipe actions, priority sort.
  6. JIRA integration. OAuth, ticket pull, dispatch sheet from a ticket, round-trip.
  7. Plan-and-dispatch (single agent). Plan generation, plan review surface, inline edits.
  8. Second-opinion mode. Layer the critic agent on top of plan-and-dispatch; build the critique notes UI.
  9. Parallel dispatch + comparison view.

Step 3 is the make-or-break. Honest confidence calibration ("when I say HIGH, I mean it") is part engineering, part prompt research. Budget 2 weeks. If it fails, the wedge is in doubt and the rest of the plan changes.

Risks

Biggest risk

The form factor is theoretically compelling and could test undecided in practice. Validation: does a design partner who tried it once prefer it over scrolling GitHub for the next agent PR? If they don't, unprompted, by week two — the wedge is wrong, and we want to know before we polish the rest.

  • Recommendation calibration drift. If "HIGH confidence" doesn't predict "reviewer accepts without changes ≥80% of the time," trust erodes fast. Continuous validation against design-partner data is mandatory, not optional.
  • Lazy critic. A second-opinion mode where the critic produces "looks good to me" is worse than no critique — it gives false confidence. Prompting must force the critic to either find something concrete or explicitly state "no concerns" (and that explicit statement should be rare enough to mean something). Measure: % of critique runs that produce ≥1 adopted note. If under ~40%, the critic is being lazy.
  • Vendor encroachment. Cursor Glass, Atlassian Rovo, Claude Dispatch, Codex Mobile all play in adjacent territory. Loupe survives only if the cross-harness review surface is meaningfully better than any vendor's single-harness surface.
  • JIRA Cloud workflow variance. Different orgs configure transitions wildly differently. Default: comment only, never transition. Transitions opt-in per project.
  • BYO-key onboarding friction. Two API keys before the first PR review feels heavy. Mitigation: strong first-run that shows one magic-moment review before asking for the second key.
  • Cost estimate trust. Two wrong estimates and users stop trusting the wallet. Be honest with ranges, sample size in tooltip.

Open questions

  • Surface: mobile-first or web-first? This PRD assumes mobile-first. Web-first would simplify dispatch and JIRA + lose the mobile wedge. Decide before code lands.
  • Recommendation generation: structured prompt, post-processing, or hybrid? Likely hybrid. 1-week spike on real PRs from Claude Code and Codex.
  • Reject-with-constraint plumbing. Both harnesses support session resume; less clear how rejection rationale lands as first-class context vs another message.
  • Diff viewing on mobile. v1: ordered hunk list. v1.1: spatial map. Long diffs: delegate to GitHub mobile.
  • Second-opinion defaults. Should the toggle default ON for tickets the user marks open-ended, OFF for tickets they mark routine? Or stay always-off and let the user opt in? Test in design-partner data.

Success metrics at v1

  • 10 design-partner engineers actively using Loupe for ≥3 ticket-→-PR cycles per week.
  • Median time-to-decision on a routine agent PR drops below 90 seconds.
  • ≥30% of reject-with-constraint actions yield an accepted PR on the next run, without re-explaining context.
  • At least one design partner spontaneously says: "this is where I review my agent PRs now."
  • Agent recommendation calibration: HIGH → ≥80% accept-without-changes; MED → 50–80%; LOW → flagged for full human read.
  • Second-opinion adopt rate: ≥40% of critique runs produce ≥1 adopted note (else the critic is being lazy).
